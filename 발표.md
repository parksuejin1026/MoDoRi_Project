# 🎓 UniMate: AI 기반 대학 학칙 비서 & 커뮤니티 프로젝트 보고서

## 1\. 프로젝트 한눈에 보기 (Overview)

  * **프로젝트명**: UniMate (유니메이트)
  * **한줄 소개**: 복잡한 학칙 및 학교 정보 검색은 AI에게 맡기고, 학교 생활은 학우들과 나누는 대학 생활 올인원 플랫폼
  * **해결하고자 한 문제 (Problem Statement)**:
      * **정보 파편화**: 학교 홈페이지 구석에 숨겨진 학칙(PDF, HWP)및 정보들을 찾기 어렵고, 모바일에서 보기 불편함.
      * **해석의 어려움**: 딱딱한 법률 용어로 된 학칙을 내 상황(휴학, 전과, 졸업 요건 등)에 맞춰 해석하기 힘듦.
  * **주요 타깃 사용자 (Persona)**:
      * 졸업 요건이나 휴학 규정이 궁금한 **대학교 3\~4학년**
      * 학교 생활 꿀팁과 동기들과의 소통이 필요한 **신입생(1학년)**
  * **기획 배경 (Why now?)**:
      * 단순 키워드 검색(Search)의 시대를 넘어, LLM(거대언어모델)을 통해 맥락을 이해하고 \*\*답변(Answer)\*\*을 주는 시대 도래.
      * 정형화된 데이터(학칙)와 비정형 데이터(커뮤니티 글)를 융합하기에 RAG(검색 증강 생성) 기술이 최적기임.

## 2\. 프로젝트 목표 & 기대 효과

  * **정량 목표 (KPI)**:
      * **학칙 검색 시간 단축**: 평균 10분 소요 → **30초 이내** 단축
      * **답변 정확도**: RAG 기반 답변 관련성 **90% 이상** 달성
  * **정성 목표**:
      * 행정실에 전화하지 않고도 궁금증을 해결하는 **Self-Service 경험** 제공.
      * 모바일 친화적 UI를 통한 **접근성(Accessibility) 개선**.
  * **생성형 AI 차별점**:
      * 기존 검색: "휴학" 검색 → 휴학 규정 파일 다운로드 → 직접 정독.
      * **UniMate**: "나 이번 학기 끝나고 군 휴학하려면 언제 신청해야 해?" → **날짜와 절차를 요약해서 답변.**

## 3\. 사용자 조사 및 인사이트 (User Research Insight)

  * **사용자 조사 방식**: 대학생 커뮤니티(에브리타임 등) 게시글 분석 및 경쟁 서비스(학교 공식 챗봇등) 분석.
  * **핵심 Pain Point**:
    1.  "학교 공식 챗봇은 '휴학'이라고 치면 그냥 홈페이지 링크만 던져줘요." (맥락 이해 부족)
    2.  "학칙 파일이 너무 길어서 내가 원하는 내용이 몇 페이지에 있는지 모르겠어요." (정보 과부하)
  * **AI 적용 필요성 인사이트**:
    1.  **개인화**: 학생마다 전공, 학년이 다르므로 일반적인 규정이 아닌 **'내 상황'에 맞는 해석**이 필요함.
    2.  **즉시성**: 궁금한 순간 바로 정답을 얻고자 하는 도파민 중심의 모바일 사용 패턴.
    3.  **요약 능력**: 수십 페이지의 문서를 3줄로 요약해 주는 LLM의 능력이 필수적.

## 4\. 솔루션 구조 (Solution Architecture)

### (1) 전체 서비스 흐름

  * **To-Be 사용자 여정**:
    1.  학교 선택 및 로그인
    2.  **[AI Chat]** "졸업 요건 알려줘" 질문 → AI가 학칙 데이터 분석 후 답변
    3.  **[Community]** "이 수업 어때?" 게시글 작성 → 학우들과 댓글 소통
    4.  **[Timetable]** 이번 학기 시간표 등록 및 관리
  * **기능 구성**:
      * **AI Rule-Look**: RAG 기반 학칙 Q\&A (Vercel AI SDK + OpenAI)
      * **MoDoRi Board**: 카테고리별 익명 게시판 (CRUD + 댓글 + 좋아요)

## 5\. 핵심 기능 시연 (Demo)

1.  **실시간 스트리밍 답변**: 채팅방에서 질문 입력 시, AI가 타자 치듯 한 글자씩 실시간으로 답변 생성 (Loading UX 최소화).
2.  **학교별 데이터 스위칭**: 설정에서 'A대학교'를 선택하면 A대학교 학칙 시트를, 'B대학교'를 선택하면 B대학교 시트를 참조하도록 동적 라우팅 구현.
3.  **모바일 최적화 게시판**: Pull-to-Refresh(당겨서 새로고침), Infinite Scroll(무한 스크롤) 등 네이티브 앱 수준의 UX 구현.

## 6\. 기술 구조 및 AI 모델 설계

  * **사용한 AI 도구 및 API**:
      * **LLM**: OpenAI `gpt-4o-mini` (속도와 비용 효율성 최적화)
      * **Framework**: Vercel AI SDK (`ai/rsc`, `ai/react`)
      * **Data Source**: Google Sheets API (학칙 데이터를 시트로 관리하여 별도 어드민 페이지 개발 비용 절감)
  * **데이터 흐름도 (Data Flow)**:
    1.  **Input**: 사용자 질문 ("이번 학기 수강신청 언제야?")
    2.  **Retrieval**: Server Action에서 Google Sheets에 저장된 해당 학교의 '학사 일정' 텍스트 데이터 로드.
    3.  **Augmentation**: [시스템 프롬프트 + 검색된 학칙 데이터 + 사용자 질문] 결합.
    4.  **Generation**: OpenAI API로 전송 → 스트리밍 방식으로 답변 출력.
  * **프롬프트 설계 예시 (System Prompt)**:
    ```markdown
    너는 {schoolName}의 친절한 학사 행정 도우미 '유니메이트'야.
    아래 제공된 [학칙 데이터]를 기반으로 학생의 질문에 정확하게 답변해줘.
    데이터에 없는 내용은 지어내지 말고 "해당 내용은 학칙에서 찾을 수 없습니다"라고 답해.
    답변은 모바일에서 보기 좋게 마크다운(Markdown) 형식을 사용해.

    [학칙 데이터]: ... (Google Sheets에서 가져온 텍스트)
    ```

## 7\. 프로젝트 결과 (Outcome) & 성과

  * **배운 점 (AI 활용 인사이트)**:
    1.  **RAG의 중요성**: LLM 단독으로는 최신/특정 학교 정보를 알 수 없음. 외부 데이터(Google Sheets) 주입이 필수적임.
    2.  **스트리밍 UX**: AI 응답 대기 시간이 2\~3초만 넘어가도 이탈률이 급증함. `StreamText` 기술로 체감 대기 시간을 0.5초로 줄임.
    3.  **비용 최적화**: 고성능 모델(GPT-4)보다 경량화 모델(GPT-4o-mini)에 명확한 Context를 주는 것이 특정 도메인(학칙)에서는 가성비가 훨씬 좋음.

## 8\. 한계점 및 개선 방향 (Roadmap)

  * **기술적 한계**: Google Sheets를 DB로 사용하여 데이터가 방대해질 경우 검색 속도 저하 우려.
  * **AI 모델의 한계**: 간혹 유사한 키워드가 있는 다른 학칙과 혼동할 수 있는 가능성 존재 (할루시네이션).
  * **Future Roadmap**:
      * **v2.0**: Google Sheets → \*\*Vector DB (Pinecone)\*\*로 마이그레이션하여 시맨틱 검색(의미 기반 검색) 도입.
      * **이미지 인식**: 시간표 이미지를 업로드하면 AI가 자동으로 텍스트로 변환하여 시간표에 등록하는 OCR 기능 추가.

## 10\. 실제 결과물 리뷰 (⭐️ AI 특화 포인트)

### "AI 없이 불가능했던 문제의 해결"

이 프로젝트의 핵심은 \*\*"CMS(콘텐츠 관리 시스템) 개발 없이 AI로 데이터 관리 효율화"\*\*를 이뤄낸 것입니다.

1.  **도구 조합 (Tool Combination)**:

      * **Text (GPT-4o-mini)** + **Database (Google Sheets)**
      * 일반적인 개발 방식이라면 학칙 데이터를 관리하는 '관리자 페이지(Admin)'를 따로 개발해야 했습니다.
      * 하지만 **Google Sheets를 백엔드 데이터 소스로 활용**하고, 이를 **LLM이 실시간으로 읽어가게 만듦**으로써, 학생회나 행정실 직원이 엑셀만 수정하면 챗봇 답변이 즉시 바뀌는 구조를 완성했습니다.

2.  **프롬프트 전략 (Prompt Engineering)**:

      * **Context Injection (맥락 주입)**: 단순히 "답해줘"가 아니라, 사용자가 선택한 '학교 코드'에 따라 **동적으로 다른 시스템 프롬프트**가 주입되도록 설계했습니다.
      * **Defense Mechanism (방어 기제)**: "모르는 것은 모른다고 해"라는 제약 조건을 걸어, AI가 거짓 규정을 만들어내 학생에게 피해를 주는 일을 방지했습니다.

3.  **생산성 향상 (Efficiency)**:

      * `ChatInterface.tsx`와 같은 복잡한 스트리밍 UI 로직을 AI 코딩 도구(Cursor/Copilot)를 활용해 **단 하루 만에 구현**했습니다.

      * 결과적으로 AI 기술을 통해 개발 기간은 단축하고, 사용자에게는 개인화된 '비서' 경험을 제공하는 **High-Tech, Low-Code** 전략을 성공적으로 입증했습니다.
